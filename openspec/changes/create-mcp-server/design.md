## Context

This document outlines the technical design for the MCP Server, a new NodeJS/TypeScript application. MCP stands for the Model-Context-Protocol, an open standard for exposing AI-usable tools (see https://modelcontextprotocol.io/docs/getting-started/intro).

The server's purpose is to dynamically generate a set of executable tools from an OpenAPI specification and expose them in an MCP-compliant format, as detailed in `proposal.md`. The server is a greenfield project with no existing architecture to consider.

## Goals / Non-Goals

**Goals:**
- To create a server that can parse any valid OpenAPI 3.x specification at startup.
- To generate a collection of in-memory tools representing the API's operations.
- To expose the generated tools via an HTTP endpoint in a format that is fully compliant with the Model-Context-Protocol.
- To design a robust configuration system for server URLs and authentication that relies on environment variables.
- To define a clear internal structure for how tools are created and how they handle parameters.

**Non-Goals:**
- To support OpenAPI 2.x (Swagger) specifications in the initial version.
- To provide a UI for interacting with the tools. This is a headless server.
- To implement hot-reloading of the OpenAPI spec without a server restart.

## Decisions

1.  **Generation Strategy: Dynamic at Startup**
    - The server will read and parse the OpenAPI specification when it starts. It will generate the tool collection in-memory.
    - **Alternative Considered:** A static code-generation step during a build process.
    - **Rationale:** The dynamic approach offers maximum flexibility. It allows the server to be pointed at different spec files or URLs without a rebuild, which is ideal for a tool designed for general-purpose API interaction.

2.  **Core Component: `ToolFactory`**
    - A central `ToolFactory` class will be responsible for the entire generation process. It will encapsulate the logic for parsing the spec, configuring the HTTP client, and building each *internal* tool object.
    - **Rationale:** This centralizes the core logic, making it easier to maintain and test, and decouples the rest of the application from the complexities of the generation process.

3.  **MCP Compliance: Adapter Layer**
    - An `McpAdapter` will be responsible for transforming the collection of *internal* tool objects generated by the `ToolFactory` into the JSON format required by the Model-Context-Protocol. This adapter will handle the mapping from our internal representation to the official MCP `tool` schema. An HTTP server (e.g., Express) will serve this JSON output on a `/tools` endpoint.
    - **Rationale:** This separates the concerns of tool *generation* (ToolFactory) from tool *presentation* (McpAdapter). This makes the system more modular and allows either part to be modified independently.

4.  **Configuration: Environment Variables with Fallbacks**
    - **Server URL:** The base URL for the target API will be sourced from the `API_SERVER_URL` environment variable. If not set, it will fall back to the first URL listed in the `servers` array of the OpenAPI spec.
    - **Authentication:** Credentials (e.g., API keys, bearer tokens) will be sourced from environment variables. The system will use a convention to map the `securityScheme` name from the spec to the variable name (e.g., `ApiKeyAuth` -> `API_KEY`).
    - **Rationale:** This follows the Twelve-Factor App methodology, separating configuration from code and keeping secrets out of the repository. It provides a clear and standard way to configure the server in different environments.

5.  **Parameter Handling: Baked-in Logic**
    - The `ToolFactory` will generate a custom `execute` function for each tool. This function will contain "baked-in" logic to map the arguments from a single `args` object to their correct locations (path, query, header, or request body) based on the operation's `parameters` definition in the spec.
    - **Rationale:** This makes the tools themselves simple to use, accepting a single object. All the complexity of HTTP request construction is handled internally by the generated function, providing a clean developer experience for the consumer of the tools.

## Risks / Trade-offs

- **Risk:** Large OpenAPI specifications (>10MB) could lead to slow server startup times due to in-memory parsing.
  - **Mitigation:** This is an acceptable trade-off for the flexibility gained. For the initial implementation, performance at startup is not a primary concern. This can be revisited if it becomes a real-world problem.
- **Risk:** The OpenAPI specification is vast, and supporting every possible feature (e.g., complex polymorphism, callbacks) could be difficult.
  - **Mitigation:** The implementation will start by supporting the most common features: basic data types, path/query/header parameters, and JSON request bodies. Support for more advanced features can be added incrementally.
